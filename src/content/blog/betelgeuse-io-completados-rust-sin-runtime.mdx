Pekka Enberg, investigador conocido por su trabajo en el kernel de Linux y coautor del paper *"I/O Is Faster Than the CPU"* (HotOS 2019), acaba de publicar en GitHub una librería de I/O para Rust que rompe con casi todas las convenciones actuales. Se llama **Betelgeuse**, llega en versión 0.1.0 con licencia MIT o Apache-2.0, y su premisa es radical: I/O asíncrona sin `async`/`await`, sin futuros, sin executor y sin tareas ocultas.

## El problema con los futuros

En Rust, la respuesta estándar a la concurrencia de red y disco es Tokio: un runtime multi-hilo con work-stealing, colas de tareas y una enorme cantidad de sincronización implícita. Para muchas aplicaciones esto funciona. Pero cuando la carga crece, los costos aparecen: cache misses por migración de tareas entre núcleos, colas de trabajo que crecen hasta consumir memoria y sobresincronización que el programador no controla directamente.

Betelgeuse parte de una premisa distinta, inspirada en el diseño de TigerBeetle: si los dispositivos de I/O modernos son tan rápidos que la CPU se ha convertido en el cuello de botella, entonces la abstracción debe ser mínima. Nada debe ejecutarse a menos que alguien lo pida explícitamente.

## Un modelo de completados

En lugar de futuros, Betelgeuse usa **completados** (*completions*). El flujo es simple:

1. El llamador prepara una operación y entrega un `&mut Completion` al backend.
2. El backend traduce la operación a una syscall (por ejemplo, `io_uring_enter` en Linux o `kevent` en macOS).
3. Cuando la operación termina, el backend deposita el resultado semántico en el mismo `Completion`.
4. El llamador revisa el slot, consume el resultado y lo recicla.

Nada avanza sin un `step()`. No hay wakeups, no hay poll, no hay runtime oculto.

Veamos cómo se ve esto con un archivo. En Betelgeuse, leer de un fichero no devuelve un `Future`; pides bytes y esperas el resultado en un slot que tú controlas:

```rust
use betelgeuse::{io_loop, Completion, CompletionResult, IO, IOFile, IOLoop, OpenOptions};
use std::path::Path;

fn leer_cabecera() -> std::io::Result<Vec<u8>> {
    let io_loop = io_loop(std::alloc::Global)?;
    let io = io_loop.io();

    let file = io.open(
        Path::new("datos.bin"),
        OpenOptions { read: true, ..Default::default() },
    )?;

    let mut completion = Completion::new();
    file.pread(&mut completion, 512, 0)?;

    // El backend no progresa solo: hay que llamar a step() explícitamente
    loop {
        io_loop.step()?;
        if let Some(result) = completion.take_result() {
            match result? {
                CompletionResult::PRead(buf) => return Ok(buf),
                _ => return Err(std::io::Error::new(
                    std::io::ErrorKind::InvalidData,
                    "resultado inesperado"
                )),
            }
        }
    }
}
```

Fíjate en la diferencia con `async`/`await`: aquí el control es total. El bucle decide cuándo avanzar el backend, cuándo revisar el completado y cuándo rendir el hilo.

## Un servidor TCP paso a paso

Para sockets, el patrón es idéntico. Cada operación —`accept`, `recv`, `send`— necesita su propio `Completion`. Un servidor mínimo mantiene un conjunto fijo de slots, nunca asigna en el camino caliente, y avanza con el clásico ciclo `server.step(); io_loop.step()`:

```rust
use betelgeuse::{io_loop, Completion, CompletionResult, IO, IOSocket, IOLoop};
use std::net::SocketAddr;

fn servidor_minimo(addr: SocketAddr) -> std::io::Result<()> {
    let io_loop = io_loop(std::alloc::Global)?;
    let io = io_loop.io();
    let socket = io.socket()?;
    socket.bind(addr)?;

    let mut accept_slot = Completion::new();
    socket.accept(&mut accept_slot)?;

    loop {
        io_loop.step()?;

        if let Some(result) = accept_slot.take_result() {
            let _cliente = match result? {
                CompletionResult::Accept(s) => s,
                _ => continue,
            };
            // Por simplicidad, cerramos inmediatamente;
            // un servidor real reciclaría el slot para recv/send.
            println!("Conexión aceptada");
        }
    }
}
```

El slot `accept_slot` es propiedad del servidor, no del runtime. Puedes tenerlo en la pila, en una estructura de estado, o en un slab preasignado. El backend nunca asigna memoria en el camino caliente.

## Bajo el capó: io_uring sin abstracciones

En Linux, Betelgeuse se apoya directamente en `io_uring`. El backend `IoUringIO` crea un anillo de 256 entradas (ajustable dinámicamente a la mitad si `ENOMEM` fuerza reducirlo), encola operaciones traduciendo `Completion` a `io_uring_sqe`, y cosecha resultados con `io_uring_enter`. No hay runtime de por medio, no hay thread pool, no hay work-stealing.

En macOS usa `kqueue` a través del backend `DarwinIO`. La interfaz pública es la misma: un `IOLoop` que entiende `step()`, un `IO` que entiende sockets y archivos y un `Completion` que el llamador posee.

Esto permite algo poco común en Rust: **deterministic simulation testing** (DST). Si reemplazas el backend nativo por uno simulado, el mismo binario de aplicación puede ejecutarse bajo tiempos controlados, órdenes de I/O prefijados e inyección de fallos, sin cambiar una línea de código de la lógica de negocio. Eso es prácticamente imposible en un modelo basado en futuros donde tareas ocultas avanzan de forma no determinista.

## Mi opinión

Betelgeuse me parece una respuesta necesaria al abismo de complejidad que Tokio puede convertirse en servidores de alta frecuencia. La decisión de eliminar `async`/`await` no es nostalgia por el código síncrono; es una apuesta deliberada por la predictibilidad. Cuando nada corre a menos que tú llames `step()`, el profiling se simplifica, los bugs de carrera desaparecen y el model checking se vuelve factible.

También valoro la restricción consciente del scope: Betelgeuse no es un framework. No resuelve framing de protocolos, no maneja buffer pools, no implementa consenso. Es una capa de I/O con un contrato claro: tú tienes los completados, el backend tiene las syscalls.

Mi principal precaución es que, en versión 0.1.0, la API aún puede moverse. La dependencia de `io-uring` 0.7.11 y `libc` 0.2 es razonable, pero la necesidad de nightly Rust—usa `#![feature(allocator_api)]`, `#![feature(coroutine_trait)]` y `#![feature(coroutines)]`—lo sitúa lejos de producción inmediata para equipos conservadores.

No obstante, como demostración de que Rust puede hacer I/O de alto rendimiento sin ceder el control a un runtime, Betelgeuse es brillante. Si alguna vez has peleado con backtraces de cuarenta frames dentro de Tokio, este diseño de "dos objetos, un bucle, cero magia" respira aire fresco.

## Conclusión

Betelgeuse no pretende reemplazar a Tokio para el noventa y nueve por ciento de los proyectos. Pero para ese uno por ciento donde cada ciclo de CPU cuenta y la no-determinación del scheduler es un enemigo, propone una alternativa austera: I/O por completados, un bucle y nada más. Cuando los fundamentos son sólidos, no hace falta magia.
