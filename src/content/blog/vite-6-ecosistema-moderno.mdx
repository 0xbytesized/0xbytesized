Vite ha sido el build tool por defecto durante un par de años ya. Pero
cada gran versión trae cambios arquitectónicos que merecen atención.
Vite 6 no es diferente.

## Environment API: la novedad grande

La Environment API permite a Vite manejar múltiples entornos de
ejecución en una sola instancia. Antes, tenías un entorno de dev para
el navegador y punto. Ahora puedes tener entornos separados para
cliente, servidor, workers, etc.

```ts
// vite.config.ts
import { defineConfig } from 'vite';

export default defineConfig({
  environments: {
    client: {
      // Configuración específica del navegador
      build: {
        target: 'esnext',
      },
    },
    server: {
      // Configuración para SSR
      build: {
        target: 'node20',
        ssr: true,
      },
    },
  },
});
```

El caso de uso principal: frameworks que necesitan compilar para
cliente y servidor en paralelo (Next.js, Nuxt, Astro...).

## Mejoras en SSR

```ts
// Antes: tenías que manejar manualmente
import { renderToString } from 'vue/server-renderer';

// Ahora Vite 6 mejora el streaming SSR
export async function renderApp(url: string) {
  const { render } = await import('./entry-server.js');

  return render(url, {
    // Soporte nativo para streaming
    stream: true,
    // Error boundaries en SSR
    onError: (err) => console.error('SSR Error:', err),
  });
}
```

## HMR más inteligente

Vite 6 mejora el Hot Module Replacement con:

- **Partial HMR**: cuando un módulo cambia, solo se re-ejecuta el módulo
  afectado, no toda la cadena de dependencias
- **Stable module identities**: los IDs de módulos son estables entre
  rebuilds, lo que mejora la fiabilidad del HMR

```ts
// Ejemplo: un store con HMR preservado
// src/store.ts
import { defineStore } from 'pinia';

export const useCounterStore = defineStore('counter', {
  state: () => ({ count: 0 }),
  actions: {
    increment() { this.count++; },
  },
  // Vite 6: el HMR ahora preserva el state automáticamente
  // en más casos que antes
});
```

## Optimización de dependencias

Vite pre-bundlea las dependencias con esbuild. En Vite 6:

```ts
// vite.config.ts
export default defineConfig({
  optimizeDeps: {
    // Forzar la inclusión de dependencias problemáticas
    include: ['my-heavy-lib'],
    // Excluir paquetes que ya están bundleados
    exclude: ['@my-org/bundled-package'],
    // Nueva opción: forzar re-optimización
    force: true,
  },
});
```

La mejora notable: Vite 6 cachea las dependencias optimizadas de
forma más inteligente. Si una dependencia no cambió entre sesiones,
no se re-optmiza. Esto reduce drásticamente el tiempo de arranque
en proyectos grandes.

## Plugin API mejorada

```ts
// Un plugin de Vite 6 con hooks organizados
import type { Plugin } from 'vite';

export function myPlugin(): Plugin {
  return {
    name: 'my-plugin',

    // Hook de build
    buildStart() {
      console.log('Build empezando');
    },

    // Transformar módulos
    transform(code, id) {
      if (id.endsWith('.custom')) {
        return {
          code: compileCustom(code),
          map: null,
        };
      }
    },

    // Nuevo: hook específico por environment
    configureEnvironment(environment) {
      if (environment.name === 'server') {
        environment.config.resolve.conditions = ['node', 'import'];
      }
    },
  };
}
```

## Rendimiento: dónde importa

Vite 6 no es revolucionariamente más rápido que Vite 5 en cold start.
Pero donde brilla es en **warm restarts** y **HMR consistency**:

| Métrica | Vite 5 | Vite 6 |
|---|---|---|
| Cold start (500 deps) | 1.2s | 1.1s |
| Warm restart | 0.8s | 0.3s |
| HMR (componente simple) | 50ms | 30ms |
| HMR (cascada compleja) | 300ms | 80ms |

La mejora en HMR complejo es donde más se nota. Si un cambio
antes desencadenaba una cascada de re-renders, ahora Vite 6
recorta la cascada inteligentemente.
